# Remote Branch Workflow — event-libs

This reference covers all procedures specific to **remote-branch-mode**
(`.aem.live` preview URL). It is loaded on demand by the main skill
when `remote-branch-mode = true`.

> **SAFETY — single rule, stated once:**
> NEVER push commits to `main`. Every push targets the feature branch
> only. Verify the branch name before every push.

> **Repository scope:**
> All branches are created on **`adobecom/event-libs`**. The `?eventlibs=`
> parameter only resolves branches on `adobecom/event-libs`. Never create
> branches on a fork repository.

---

## 1. Publishing a `.aem.page` URL

Parse the URL to extract org, repo, and page path:
```
https://main--<repo>--<org>.aem.page/<path>
```

### Path safety check (BLOCKING)

- **Path contains `/drafts/`:** safe to publish automatically.
- **Path does NOT contain `/drafts/`:** present this warning and wait for
  explicit confirmation:
  ```
  WARNING: Publishing this page will make it live on production.
  Path: <path>
  Are you sure you want to proceed? (y/n)
  ```
  Do NOT call the publish API without passing this check.

### Publish via EDS admin API

```bash
# Preview first
curl -s -w "\n%{http_code}" -X POST \
  "https://admin.hlx.page/preview/<org>/<repo>/main/<path>"

# Then publish
curl -s -w "\n%{http_code}" -X POST \
  "https://admin.hlx.page/live/<org>/<repo>/main/<path>"
```

Expect **200 OK** for both. On success, the published URL is:
```
https://main--<repo>--<org>.aem.live/<path>
```

On failure (401/403), ask the user to check permissions or publish manually.

## 2. Branch creation

All git operations target `adobecom/event-libs` via the **`origin`** remote
(or whichever remote points to `adobecom/event-libs` — verify with
`git remote -v`).

1. **Determine branch name**:
   - If the user explicitly requested a specific feature branch, use that name.
   - Otherwise, default to `<block-name>-autogenerated`
     (e.g. block name "profile cards" → `profile-cards-autogenerated`).

2. **Check for existing branch**:
   ```bash
   git ls-remote origin refs/heads/<branch-name>
   ```
   If the branch already exists, inform the user and ask how to proceed:
   - Delete and recreate it
   - Use a different name (e.g. `<block-name>-autogenerated-2`)
   - Reuse the existing branch

3. **Determine the base branch**:
   - Default: `main`.
   - If the user specified a custom base branch, check whether it exists on
     `adobecom/event-libs`:
     ```bash
     git ls-remote origin refs/heads/<base-branch>
     ```
     If it does not exist, push it:
     ```bash
     git push origin <local-branch>:refs/heads/<base-branch>
     ```

4. **Create the feature branch** from the base branch:
   ```bash
   git fetch origin <base-branch>
   git push origin origin/<base-branch>:refs/heads/<branch-name>
   ```

5. **Inform the user** which branch will receive all code changes and
   **wait for confirmation**:
   ```
   All code will be pushed to branch: <branch-name>
   Repository: adobecom/event-libs
   Base branch: <base-branch>
   Branch "main" will NOT be modified.
   Proceed? (y/n)
   ```

> **STOP**: Do NOT proceed until the user confirms the branch.

## 3. Pushing code to the feature branch

Use **git CLI** to commit and push to the feature branch on
`adobecom/event-libs`.

### Workflow

1. **Fetch and check out the feature branch** from origin:
   ```bash
   git fetch origin <branch-name>
   git checkout -b <branch-name>-work origin/<branch-name>
   ```
   Use a temporary local branch name (e.g. `<branch-name>-work`) to avoid
   conflicts with any existing local branch.

2. **Create or copy the block files** into the working tree:
   - `event-libs/v1/blocks/<name>/<name>.js`
   - `event-libs/v1/blocks/<name>/<name>.css`
   - Edit `event-libs/v1/libs.js` to add the block name to `EVENT_BLOCKS`

3. **Stage all generated files and commit**:
   ```bash
   git add event-libs/v1/blocks/<name>/ event-libs/v1/libs.js
   git add test/unit/blocks/<name>/
   git commit -m "Add <block-name> block (autogenerated)"
   git push origin HEAD:<branch-name>
   ```

4. **Clean up** the temporary local branch:
   ```bash
   git checkout <previous-branch>
   git branch -D <branch-name>-work
   ```

For subsequent pushes (visual fixes, a11y fixes), repeat the same pattern:
fetch, checkout, apply changes, commit, push, clean up.

After the push succeeds, the block will be available at:
```
https://main--<page-repo>--<page-org>.aem.live/<path>?eventlibs=<branch-name>
```

## 4. CDN force-refresh after push

The CDN and browser may cache stale JS/CSS from the feature branch. After
every push, force-refresh in Playwright.

### 4a. Trigger EDS code preview for changed files

For each file you pushed, call the EDS code preview API so the CDN picks
up the new version:

```bash
curl -s -X POST \
  "https://admin.hlx.page/code/adobecom/event-libs/<branch>/event-libs/v1/blocks/<name>/<name>.js"
curl -s -X POST \
  "https://admin.hlx.page/code/adobecom/event-libs/<branch>/event-libs/v1/blocks/<name>/<name>.css"
curl -s -X POST \
  "https://admin.hlx.page/code/adobecom/event-libs/<branch>/event-libs/v1/libs.js"
```

> **NEVER use a wildcard or broad path** for code preview.  Always reference
> the specific files that changed.

### 4b. Bypass browser HTTP cache in Playwright

```js
await page.route('**/*.{js,css}', (route) => {
  route.continue({
    headers: {
      ...route.request().headers(),
      'cache-control': 'no-cache',
    },
  });
});
```

Set this route **once** at the start of the Playwright session. It persists
across navigations and reloads.

### 4c. Reload and verify

1. Navigate to the Playwright URL (with `?eventlibs=<branch-name>`).
2. Hard-reload the page.
3. Wait for the block to fully render before proceeding.

If files still 404 after force-refresh, wait 10–15 seconds and repeat
from 4a.

## 5. Batched iteration strategy

Each verification in remote-branch-mode requires a push + CDN refresh.
To avoid excessive commits and API load, **batch multiple fixes into a
single push**:

1. After each screenshot comparison, collect all issues found.
2. Apply fixes to the local files (JS and/or CSS).
3. Before pushing, assess whether the accumulated changes are in a
   high-confidence state: correct syntax, consistent token usage, all
   breakpoints addressed, no obvious regressions.
4. Only push when the batch of fixes is coherent and likely to produce a
   meaningful visual improvement.  A single push covering 3–5 fixes is
   preferable to 5 separate pushes.
5. After pushing, repeat the force-refresh procedure (section 4), then
   take the new screenshot.

Use a descriptive commit message summarizing the batch, e.g.
`"Fix heading alignment, tablet spacing, and CTA padding"`.
